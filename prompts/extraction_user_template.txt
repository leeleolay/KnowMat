Step 1: Global metadata and role distinction

- Extract the paper title and DOI into the root `Paper_Metadata` object.
- Extract this paper's own retained materials as `Target`.
- If the current paper highlights direct benchmark specimens or direct comparison rows in its retained comparison set, you may extract them as separate `Reference` items, but never mix them into a `Target` item.
- If the paper has direct comparison tables/figures/literature benchmark rows that belong to the retained comparison set, each cited comparison alloy/state must be output as its own `Reference` item.
- Do not create `Reference` items from broad literature landscape plots, review-style alloy lists, umbrella labels such as `various AM alloys`, or generic superiority claims unless the current paper makes them explicit sample-resolved comparator labels.

Step 2: Item decomposition

- One item = one unique combination of composition + process.
- Before emitting JSON, enumerate all explicit specimen/sample labels from nomenclature tables, processing tables, property tables, captions, and comparison rows. Internally classify each label into exactly one of: `KEEP_TARGET`, `KEEP_REFERENCE`, or `SKIP_NON_ITEM`.
- Track the inventory count explicitly. Final item count should equal the number of labels assigned to `KEEP_TARGET` + `KEEP_REFERENCE`. If final items are fewer, revisit missing labels; if final items are greater, look for alias duplicates or fake items promoted from background text.
- Split only when the paper treats the difference as specimen-defining: composition, consolidated route, explicit retained sample/state label, build/loading orientation, or a final reported treatment/parameter cohort with its own aligned process, microstructure, or property evidence.
- Do not split into new items for test temperature, test stress, creep load, strain rate, dwell time, corrosion environment, or replicate index. Those stay inside `Properties_Info` for the same specimen.
- Do not automatically split every screening composition, laser-power sweep, aging-time sweep, or exploratory process setting. If those candidates only support selection of one optimized specimen and are not carried forward as enduring specimens in the main comparison set, classify them as `SKIP_NON_ITEM`.
- Give each item a stable `Sample_ID`; all tests that belong to the same specimen must share that id.
- One explicit specimen should correspond to at most one item. If the paper uses both a shorthand code and a long descriptive name for the same specimen, merge them into one item with one canonical `Sample_ID`.
- If the paper defines an abbreviated sample nomenclature table and then reports direction/state-specific rows (for example `3090` in the process table and `B3090` / `S3090` in the property table), you must output one item per explicit row/state.
- If one alloy appears in multiple retained explicit states such as as-built, annealed, aged 8 h, aged 48 h, heat-stabilized, solutionized, or different build/loading directions, output one item per explicit state even when chemistry is unchanged.
- `KEEP_TARGET`: this paper treats the label/state as a real retained specimen with sample-resolved process, microstructure, or property evidence.
- `KEEP_REFERENCE`: this paper gives explicit comparison values/rows/caption/legend entries for that benchmark specimen, and that comparator is a specific label/state rather than a broad literature family.
- `SKIP_NON_ITEM`: background literature mentions, generic route descriptions, unnamed process fragments, or repeated aliases of an already-kept specimen.
- If the paper contains a final comparison matrix, final property table, or headline result set, align your item inventory to that retained comparison set first. Do not let upstream screening branches or auxiliary optimization candidates outnumber the final compared specimens unless the paper explicitly keeps them as standalone results specimens.
- Important exception: if the paper's central contribution IS a process-state or recycle-state matrix, then that whole matrix is the retained comparison set. Do not collapse explicit rows such as `R0/R1/R5/...`, `S0/S5/S10/...`, `as-fabricated/T6`, `single-aged/double-aged`, or similar named state chains to one representative item.
- If the retained comparison set contains directly characterized upstream or process-comparison endpoints such as powder vs `PBF-LB` vs `PBF-EB`, optimized ingot vs `LPBF` states, or paired recycle families `R*` and `S*`, keep every explicit retained endpoint family/state that has sample-resolved data.
- If the retained comparison set contains state-paired or label-paired benchmarks such as `2195` as-fabricated vs `2195` T6, `Ni3Al` vs `TiAl`, or multiple named literature alloys in one legend, keep one item per explicit benchmark label/state. Do not collapse several distinct benchmarks into one umbrella reference.
- When only a subset of labels survives into the retained comparison set, do not invent additional intermediate states from narrative-only mentions. If final retained items are `R1` and `R5`, do not create `R2` or generic `Powder` items unless they are also explicit sample-resolved endpoints.
- For gradient materials, prefer one item with `Gradient_Material = true` unless the paper explicitly reports layer-specific independent data.
- Only split a gradient specimen into multiple items when independent per-layer or per-step data is explicit. In that case, keep `Gradient_Material = true` and link all split items with the same `Gradient_Group_ID`.
- If the same item has multiple testing conditions, keep one item and place all tests in that item's `Properties_Info`.
- If a variant is explicitly named but only sparse evidence exists, keep a sparse item only when that variant is retained in the paper's actual specimen comparison set.
- When the paper contains a multi-step route such as casting/ingot screening -> powder preparation -> AM, keep only the upstream states that the paper itself retains as specimen-level comparison endpoints.
- Feedstock/powder/ingot/cast/recycled-powder/intermediate states should become standalone items only when the paper itself characterizes or tabulates that state as a specimen or uses it as an explicit comparison endpoint. If the powder has its own phase/composition/microstructure data, keep it; if it is only preparation history, keep it in notes/process text.
- Distinguish retained upstream endpoints from precursor history: a laser-glazed optimized ingot, powder-recycle state, or as-fabricated vs T6 specimen can be a valid retained item; a planar-flow-cast precursor or spray-deposited precursor mentioned only as background inspiration should usually be `SKIP_NON_ITEM`.

Step 3: Composition extraction (`Composition_Info`)

- Fill `Role`, `Alloy_Name_Raw`, `Nominal_Composition`, and `Measured_Composition`.
- Keep nominal and measured composition strictly separate.
- Record special composition annotations such as ranges, upper limits, or impurity notes in `Composition_Info.Note` / runtime `composition_note`.
- `nominal_composition` and `measured_composition` must be flat element->number maps only; never use step keys like `{"0": {...}, "5": {...}}`. If the paper truly reports explicit layer-specific graded data, create separate composition entries instead.
- Faithfully reflect the reported composition values. Do not renormalize just to force a 100.0 total; only fill a named balance element or justified residual `other` when the source supports it.
- Use explicit source cues to separate provenance: design / nominal / target / supplier formula belongs in `nominal_composition`; ICP-OES / EDS / EPMA / measured tables belong in `measured_composition`.
- Preserve the original unit exactly as reported (`wt%` vs `at%`). Never convert unless the paper explicitly performs that conversion.
- If a specific element is marked `Bal.` / `balance` / `remainder`, compute that named element as `100 - sum(other explicit elements)` and store it under that element symbol.
- `other` is only for unresolved trace impurities or residual minor species that are not individually listed. Never use `other` for a known balance element or for a bundle of named elements.
- If `other` would become the dominant matrix fraction and the alloy system clearly implies a base element, store that base element explicitly instead of leaving it as `other`.
- Preserve mature commercial alloy names with null normalized arrays if no explicit numeric composition is reported.
- Never convert trade names or polymer abbreviations into fake element formulas.

Step 4: Process extraction (`Process_Info`)

- Build `Process_Text.original` and `Process_Text.simplified`.
- Extract explicit equipment information into `Process_Info.Equipment` / runtime `equipment`.
- Keep `process_category` aligned to the base manufacturing route, not only the latest post-treatment. If the sample is AM + heat treatment, preserve the AM base route and put the post-treatment into `Process_Text` / `Key_Params`.
- `process_category` is for the consolidated specimen route, not feedstock-only operations. Powder atomization, powder drying, powder characterization, or powder preparation inside an AM paper do not make the final specimen `Powder_Metallurgy`.
- Extract quantified variables into `Key_Params`.
- Use canonical parameter names where possible, and dynamic keys with unit hints when needed.
- Preserve heat-treatment ranges as strings such as `1273-1373` or `2-4`; never average them.
- For multistage processing, use stage suffixes such as `_Stage1_K` and never average stages.
- Prefer extracting process parameters from Methods / Experimental / Processing sections and process tables before relying on discussion text.
- Treat `LAAM`, `laser-aided additive manufacturing`, `laser aided additive manufacturing`, `laser cladding`, and `laser metal deposition` as `AM_DED` when they describe deposition.
- For `PBF-EB`, `electron beam powder bed fusion`, `electron beam melting`, or `Arcam` routes, use `EBM` and store beam-specific parameters such as `Beam_Current_mA` and `Acceleration_Voltage_kV` rather than `Laser_Power_W`.
- When the specimen is AM plus explicit aging/annealing/thermal stabilization/stress relief, keep the AM base route and retain the post-treatment in the same item.

Step 5: Microstructure extraction (`Microstructure_Info`)

- Build `Microstructure_Text.original` and `Microstructure_Text.simplified`.
- `Main_Phase` must describe the matrix phase. Do not label `Laves`, carbides, or gamma-prime as the main phase.
- If the matrix is explicitly dual-phase or multi-phase, combined labels such as `FCC + BCC` or `FCC + L12` are allowed only when both are reported as base matrix constituents.
- Put explicit precipitate types into `Precipitates[]` / runtime `precipitates`.
- Extract `Main_Phase`, `Porosity_pct`, `Relative_Density_pct`, `Grain_Size_avg_um`, `Precipitate_Size_avg_nm`, and `Precipitate_Volume_Fraction_pct` whenever evidence exists.
- `Relative_Density_pct` must stay null when not reported. Never infer 100%.
- Put advanced quantified microstructure metrics under `Advanced_Quantitative_Features`.
- When equiaxed vs columnar, FG vs CG, interior vs wall, or similar sub-regions are reported separately, keep each quantitative value as its own `Advanced_Quantitative_Features` key instead of averaging them into one grain-size field.
- `Advanced_Quantitative_Features` is quantitative-only: keep numeric values/ranges such as dislocation density, KAM, lattice parameter, phase fraction, or sub-region grain size; do not place `SEM/TEM/XRD/EBSD/APT` narrative descriptions there.

Step 6: Properties extraction (`Properties_Info`)

- Extract all reported properties for the same item.
- Keep strict ordering from Tier 1 to Tier 3.
- Use `Elongation_Total`, `Elongation_Uniform`, and `Elongation_At_Fracture` instead of the older generic elongation field.

- Use compressive property names only for compressive contexts.
- `Test_Temperature_K` is only the actual testing environment temperature, not a process temperature.
- Also extract `Strain_Rate_s1`, `Tensile_Speed_mm_min`, `Hardness_Load`, `Hardness_Dwell_Time_s`, `Data_Source`, `Test_Specimen`, and `Note` when supported.
- For ranges, keep midpoint in `Value_Numeric` and source range in `Value_Range`.
- For mean±std, keep mean in `Value_Numeric` and std in `Value_StdDev`.
- For hardness/microhardness, if unit text is omitted but context indicates Vickers hardness, set `Unit` to `HV`.
- Enumerate every explicit test temperature, stress level, hold time, orientation, and composition/parameter level. Do not keep only end members or one representative temperature when intermediate rows are explicitly reported.
- For creep / stress-rupture / high-temperature property matrices, capture every reported temperature-stress-condition combination as separate property entries under the aligned item.
- Never convert a test-condition matrix into extra items. Multiple test temperatures, creep stresses, exposure times, or loading conditions for the same specimen must stay inside one item unless the paper explicitly says the specimen itself changed before testing.
- If a property is explicitly reported in a labelled figure, caption, or readable plotted data source, you may set `Data_Source = image`; never invent unlabeled plot values.
- Use `Hardness_HV` only for Vickers-hardness context. For nanoindentation or instrumented indentation hardness reported in `GPa`, use a dynamic property name such as `Nanoindentation_Hardness` with unit `GPa` instead of relabeling it as `Hardness_HV`.
